# MiMo Harness 待办事项

## 开发循环流程（每个任务必须遵循）

```
┌─────────────────────────────────────────────────────────────────┐
│                        开发循环流程                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌──────────┐     ┌──────────┐     ┌──────────┐                │
│  │ 1.实现功能 │────▶│ 2.添加测试 │────▶│ 3.测试分析 │               │
│  └──────────┘     └──────────┘     └──────────┘                │
│       ▲                                    │                    │
│       │                                    ▼                    │
│  ┌──────────┐     ┌──────────┐     ┌──────────┐                │
│  │ 6.改进代码 │◀────│ 5.Code    │◀────│ 4.改进代码 │               │
│  │   和测试   │     │  Review  │     │          │                │
│  └──────────┘     └──────────┘     └──────────┘                │
│       │                                                         │
│       └─────────────────────────────────────────────────────▶  │
│                        重复直到无重大漏洞                        │
└─────────────────────────────────────────────────────────────────┘
```

### 循环步骤详解

| 步骤 | 名称 | 具体要求 |
|------|------|----------|
| 1 | 实现功能 | 编写核心功能代码，遵循项目架构 |
| 2 | 添加测试 | 编写精简测试，只测试主要根函数 |
| 3 | 测试分析 | 运行测试，分析结果，识别问题 |
| 4 | 改进代码 | 根据测试结果修复问题，优化实现 |
| 5 | Code Review | 客观审查代码，寻找漏洞和改进点 |
| 6 | 改进代码和测试 | 根据 review 结果修复问题 |

### 终止条件

循环在以下情况结束：
- 连续 2-3 次迭代无重大漏洞
- 测试通过率稳定在 95% 以上
- 无新增的安全或性能问题

---

## 测试要求

### 真实 API 测试
- **优先使用真实 API**: 尽量使用 MiMo API 进行端到端测试
- **压力测试**: 模拟真实环境下的并发和负载
- **错误场景**: 测试 API 限流、超时、网络错误等异常情况
- **成本控制**: 合理控制测试成本，避免过度调用

### 函数测试原则
- **精简测试**: 只测试功能的主要根函数，不测试内部实现细节
- **边界条件**: 测试输入边界、异常输入、空值等情况
- **依赖隔离**: 使用 mock 隔离外部依赖（API、文件系统等）
- **可重复性**: 测试结果应可稳定复现

### 测试覆盖目标
- 核心功能：100% 覆盖
- 边界条件：90% 覆盖
- 错误处理：80% 覆盖

---

## Code Review 要求

### 审查原则
- **项目定位**: 此项目为个人实践项目，非生产级系统
- **客观评价**: 保持客观，不夸大问题严重性
- **务实建议**: 给出可操作的改进建议，而非理论性批评
- **安全优先**: 重点关注安全漏洞和数据泄露风险

### 审查范围
- **功能正确性**: 代码是否正确实现了预期功能
- **安全漏洞**: 是否存在注入、越权、数据泄露等风险
- **性能问题**: 是否有明显的性能瓶颈或资源浪费
- **代码质量**: 可读性、可维护性、错误处理

### 审查限制
- **不接入服务器**: 不要求部署到生产环境
- **不接入外部系统**: 不要求集成第三方服务
- **个人项目标准**: 按照个人项目标准评估，非企业级标准

### 审查输出格式
```markdown
## Code Review 结果

### 严重问题 (必须修复)
- [ ] 问题描述 + 修复建议

### 改进建议 (建议修复)
- [ ] 问题描述 + 改进建议

### 优化空间 (可选优化)
- [ ] 问题描述 + 优化建议
```

## 1. 优化 Tokens 计数方式

**目标**: 参考最新的 Claude Code 和 Codex 的实现方式，优化 tokens 统计和计数机制

**当前状态**:
- 使用加权启发式算法（~3.5-4.5 chars/token）
- 基于内容类型的差异化比率（tool/system/code/natural language）

### 开发循环

#### 迭代 1: 基础实现
1. **实现功能**
   - [ ] 研究 Claude Code/Codex 的 token 计数实现
   - [ ] 实现基于 tiktoken 的精确计数
   - [ ] 添加流式输出时的实时统计

2. **添加测试**
   - [ ] 测试 `estimate_tokens()` 函数的准确性
   - [ ] 测试不同类型内容的 token 计数
   - [ ] 测试流式输出的统计累积

3. **测试分析**
   - [ ] 对比 tiktoken 计数与启发式计数的差异
   - [ ] 分析性能影响（计数开销）
   - [ ] 识别边界情况和异常

4. **改进代码**
   - [ ] 优化计数性能（缓存、批量处理）
   - [ ] 处理边界情况（空消息、超长内容）

5. **Code Review**
   - [ ] 审查计数逻辑的正确性
   - [ ] 检查内存使用和性能
   - [ ] 验证与 Claude Code 的一致性

6. **改进代码和测试**
   - [ ] 根据 review 修复问题
   - [ ] 补充遗漏的测试用例

#### 迭代 2: 高级功能
1. **实现功能**
   - [ ] 实现 Token 预算管理和警告机制

   - [ ] 测试统计报告的准确性
   - [ ] 测试压缩估算的合理性

3. **测试分析**
   - [ ] 验证预算管理的有效性
   - [ ] 分析统计报告的实用性

4. **改进代码**
   - [ ] 优化预算管理策略
   - [ ] 改进统计报告格式

5. **Code Review**
   - [ ] 审查预算管理的完整性
   - [ ] 检查统计报告的准确性

6. **改进代码和测试**
   - [ ] 根据 review 优化实现
   - [ ] 完善测试覆盖

**终止条件**:
- Token 计数误差 < 5%
